# FuckBilibiliComments

一个强大的 B 站评论爬取与分析工具，支持多种爬取模式、基础数据清洗处理，以及一套完整的 **Windows 桌面 GUI**。

## 📋 基础 Q&A

### 这个工具能做什么？

**FuckBilibiliComments** 是一个 B 站评论数据采集与分析工具，具备以下核心功能：

- **多模式评论爬取**：基于 BV 号或视频链接，识别视频并爬取评论，支持充电视频评论与"楼中楼"评论；**单次爬取上限在 20000 条左右**（原因详见进阶 Q&A）
- **智能去重处理**：自动识别并处理重复评论，只保留爬取时间最新的评论
- **基础数据分析**：按分钟 / 小时 / 日 / 月 / 年统计评论数量并生成趋势折线图
- **评论可视化**：对回复数超过 5 条的主楼生成"楼中楼拖尾图"，直观展示评论层级关系
- **CSV 去重合并**：将两次不同时间或不同账号的爬取结果合并去重，提高覆盖率
- **历史记录管理**：记录每次爬取任务，支持一键打开输出文件夹、查看日志占用、批量清理

### 关于"主楼"和"楼中楼"

- **主楼**：直接回复视频的一级评论
- **楼中楼**：点击"回复"按钮、针对某条评论进行的二级回复

## 🔑 前置准备：获取 Cookie 与 User-Agent

在开始爬取之前，你需要先准备好 B 站账号的 **Cookie** 和 **User-Agent**，并在"账号管理"Tab 中录入。

### 获取 Cookie

**Cookie 是什么，为什么需要它？**

当你在 B 站登录后，浏览器会把登录状态以 Cookie 的形式缓存在本地，此后每次向 B 站发送请求时都会自动附带这个 Cookie，B 站通过它来识别"这个用户已经登录了"。B 站对未登录用户限制了只能查看前几条评论；本工具通过在每次 API 请求中附带你的 Cookie，模拟已登录用户的浏览行为，从而获取完整的评论数据。

**如何获取 Cookie？**

推荐使用 **Cookie-Editor** 浏览器扩展，访问官网 [https://cookie-editor.com/](https://cookie-editor.com/) 根据你的浏览器选择对应的下载链接安装（**请务必通过官网进入商店安装，不要在扩展商店中自行搜索**，搜索结果中存在仿冒插件）。

安装后，登录 B 站，点击浏览器工具栏中的 Cookie-Editor 图标，在弹出面板底部点击 **Export（导出）**，Cookie 字符串将自动复制到剪贴板，粘贴到账号管理的 Cookie 输入框即可。

**脚本会自动过滤哪些 Cookie 字段？**

浏览器导出的 Cookie 通常包含几十个字段，本工具只提取 B 站 API 真正需要的核心字段，其余字段在保存时自动丢弃：

| 字段名 | 作用 |
|--------|------|
| `SESSDATA` | **登录身份令牌**（最核心）。B 站通过此字段识别已登录用户，缺少则只能看到少量评论 |
| `bili_jct` | CSRF 防跨站请求伪造令牌，B 站 API 校验请求合法性时使用 |
| `DedeUserID` | 登录用户的 B 站 UID（数字），标识当前账号 |
| `DedeUserID__ckMd5` | UID 的校验哈希，与 `DedeUserID` 配套使用 |
| `buvid3` | 设备指纹（浏览器级别），B 站用于识别访问设备 |
| `buvid4` | 设备指纹的增强版本，含更多熵值 |
| `buvid_fp` | 设备指纹的哈希摘要 |
| `buvid_fp_plain` | 设备指纹的明文形式（部分账号有此字段） |
| `b_nut` | 设备 Cookie 的首次生成时间戳 |
| `_uuid` | 设备唯一标识符（UUID 格式） |
| `sid` | 当前会话 ID |
| `b_lsid` | 本地存储会话 ID |
| `bili_ticket` | 短期访问凭证（JWT 格式），用于部分接口鉴权 |
| `bili_ticket_expires` | `bili_ticket` 的过期时间戳 |
| `fingerprint` | 浏览器指纹（部分账号有此字段） |
| `rpdid` | 评论区请求追踪 ID |

以上字段之外的其他 Cookie（如 `theme_style`、`home_feed_column` 等界面偏好字段）不会被保存，不影响爬取功能。

### 获取 User-Agent

User-Agent 用于告诉 B 站"这个请求来自哪种浏览器"，可通过以下任一方式获取：

- **Chrome**：地址栏输入 `chrome://version`，找到"用户代理"一行
- **Edge**：地址栏输入 `edge://version`，找到"用户代理"一行
- **Firefox**：地址栏输入 `about:support`，找到"用户代理"一行
- **通用方法**：按 F12 打开开发者工具 → Network 标签页 → 刷新页面 → 点击任意请求 → 在 Request Headers 中找到 `User-Agent` 字段，复制

建议填写与你 Cookie 来源相同的浏览器的 UA，保持请求特征一致。

## ⚙️ 工作原理

### 脚本在做什么？

本工具的本质是**模拟已登录用户在浏览器里翻看评论的行为**。每次你在 B 站滚动评论区，浏览器就会向 B 站 API 发送一次 HTTP 请求，B 站返回当页的评论数据（JSON 格式）。本工具通过 Python 的 `requests` 库不断重复这个过程，直到所有评论页都被爬取完毕。

### 发给 B 站的请求包含什么？

每次请求由**请求头**和**请求参数**两部分组成：

**请求头（模拟浏览器身份）：**

| 字段 | 内容 | 作用 |
|------|------|------|
| `Cookie` | 你的 B 站登录 Cookie | 告诉 B 站"我已登录，可以查看全部评论" |
| `User-Agent` | 浏览器标识字符串 | 告诉 B 站"我使用的是某某浏览器" |
| `Referer` | `https://www.bilibili.com/` | 告诉 B 站"这个请求来自 B 站页面" |

**请求参数（告诉 B 站要哪个视频的第几页）：**

| 参数 | 示例值 | 含义 |
|------|--------|------|
| `oid` | `115066334743950` | 视频的 AV 号（av id），唯一标识视频 |
| `type` | `1` | 资源类型，1 代表普通视频 |
| `sort` | `0` / `1` | 排序方式：0 = 热度排序，1 = 时间排序 |
| `pn` | `1`, `2`, `3`… | 页码（Page Number） |
| `ps` | `20` | 每页评论数（Page Size，固定 20） |

### B 站返回什么？

B 站以 JSON 格式返回当页评论列表，每条评论包含以下关键字段：

| JSON 字段 | 对应输出列 | 含义 |
|-----------|-----------|------|
| `rpid` / `rpid_str` | rpid | **评论唯一 ID**（Reply ID）；这是去重的核心依据 |
| `parent` / `parent_str` | parent | 被回复评论的 rpid；主楼评论此值为 0 |
| `content.message` | 评论内容 | 评论正文 |
| `member.uname` | 用户名称 | 评论者昵称 |
| `member.mid` | 用户 ID | 评论者 UID |
| `member.level_info.current_level` | 用户等级 | 评论者 B 站等级（0–6） |
| `member.sex` | 性别 | 评论者性别（B 站注册时填写） |
| `like` | 点赞数 | 该评论的获赞数量 |
| `rcount` | 回复数 | 该评论的楼中楼回复数量 |
| `ctime` | 时间戳 / 发布时间 | 评论发布的 UNIX 时间戳（秒）；工具将其转换为可读时间格式 |
| `reply_control.location` | IP 地区 | 评论者发布时的 IP 属地（如"IP属地：广东"） |

### rpid 和 parent 有什么用？

- **rpid** 是每条评论在该视频下的永久唯一标识符，不随点赞数、回复数的变化而改变。两条 rpid 相同的评论即为同一条评论——这是去重的唯一依据。
- **parent** 指向"这条评论回复的是哪条评论"。通过逐层匹配 parent → rpid，可以还原出完整的评论层级树，用于生成楼中楼拖尾图。

## 🔬 进阶 Q&A

### Q：为什么单次爬取的评论数量存在上限？

A：经过多次测试，发现在爬取到大约 10000 条评论时，B 站 API 会返回类似 `Connection aborted` 的错误，强行中断爬取。

因此，笔者设计了综合模式——先热度爬取、再时间爬取；它理论上可获取 20000 条左右，但因两次爬取存在大量重叠，**保底能获取约 15000 条不重复评论**，超过 20000 条的情况并不多见。

笔者个人猜测这是 B 站服务器的反爬机制之一——因为它的操作本质上是在“模拟”你滑动评论区查看评论的操作，B站每次滑动会刷新20条评论，而爬取到10000条时意味着你已经滑动刷新了500次，如果您有闲心的话可以试一下刷新500次后浏览器顶不顶得住。

### Q：评论数超过 2 万条时怎么办？

A：这就是**迭代模式**的用武之地——反复爬取 + 多次合并去重来突破单次上限。

首先需要说明的是，当评论数已经超过2万条时，仅靠模拟请求是不可能完整爬取的；但是，对于一些视频（比如二游重大版本的PV），我们是可以预见它的评论数会超过这个阈值的。这样子，我们实际上可以在它发布后开始反复爬取评论，剔除重复评论，并在评论数增长逐渐放缓后得到一份覆盖率基本完整的评论csv文件的。

但是，如何判断评论数增长已经“放缓”了呢？笔者在迭代模式中提供了两种判断模式：

- **限定时间模式**：设定一个时间窗口（如 24 小时），在此期间持续爬取
- **限定重复率模式**：当相邻两次爬取的重复率超过阈值时自动停止

当然，这意味着迭代模式会在本地积累大量数据文件（可能达到GB级别），所以请提前确保硬盘空间充足。

### Q：脚本如何判断两条评论是否重复？

A：通过 B 站评论的 **rpid 字段**（Reply ID，评论唯一标识符，在正常浏览评论时不显示）。只要两条评论的 rpid 相同，无论点赞数、回复数如何变化，都视为同一条评论进行去重。这也是为什么 CSV 去重工具要求输入含有 rpid 列的原始数据文件，而非已移除该列的最终输出文件。

### Q：楼中楼拖尾图是如何生成的？

A：每条评论除了 rpid 外，还携带 **parent 字段**，表示"这条评论是对哪条评论的回复"（parent 值等于被回复评论的 rpid）。通过逐层匹配 parent-rpid，可以还原出完整的评论树结构，再通过 PIL 库绘制成带颜色层级的可视化长图。对回复数超过 5 条的主楼自动触发生成。
