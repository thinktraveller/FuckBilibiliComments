### 3、找到cookie和user_agent

如果你是为了配置 cookie，最省事的方式是直接安装浏览器扩展 [Cookie-Editor](https://cookie-editor.com/)，然后在 B 站页面里导出 Header String 或完整 cookie 字符串。

当前脚本已经兼容 Cookie-Editor 导出的 Header String / cookie 字符串格式，不需要手动去拼请求头。

然后，你需要查看自己当前浏览器的 User-Agent，可以用下面这些方法：

- **Chrome**：在地址栏输入 `chrome://version`，页面里能找到 `User Agent` / `用户代理`
- **Edge**：在地址栏输入 `edge://version`，页面里能找到 `User Agent` / `用户代理`
- **Firefox**：在地址栏输入 `about:support`，页面里能找到 `User Agent` / `用户代理`

到这里，您可能初步明白本脚本的运作原理了：当我们在B站点开一个视频、看到一条评论时，**本质上是通过官方API、向B站发送了一个请求网址，而在对应的响应信息里包含了评论内容、用户名称、等级、点赞数、发表时间等等信息；**

而您如果有在陌生设备上登陆B站的经历的话，可能也知道**在B站如果不登录，是只能看到视频的前几条评论的**；而我们之所以不用每次打开B站都要登录一次，是因为浏览器通过**cookie**缓存了我们的信息，在每次访问B站发送请求网址时，把这个cookie一起发过去，告诉B站“这个用户已经登录了！Ta可以看到所有评论！”

这就是为什么脚本要求您输入 cookie 和 user_agent——cookie 用于模拟用户的登录状态，user_agent 用于模拟用户所使用的浏览器，这个脚本的本质就是**模拟**用户的正常浏览行为，不断地发送带 cookie 和 user_agent 的请求网址，来让 B 站以为这个用户在不断翻看评论；

也正是因此，**每个账号都有自己专属的 cookie**，如果爬取过于频繁可能导致 B 站 API 暂时封禁该 cookie（表现为脚本运行时的 412 错误）；虽然在长期的开发和测试阶段没有发现因为该错误而对正常浏览行为作出的限制，**但笔者仍然不保证过分频繁的爬取会不会影响账号的正常使用，也不建议和他人共享 cookie 信息！**

## 🔬 进阶Q&A

### Q: 为什么单次爬取的评论数量存在上限？

A：经过多次测试，发现在爬取到大概10000条评论时，会返回诸如'Connection aborted'等错误，强行停止爬取（说白了就是发送给API的请求没有得到正常响应）；笔者个人猜测这是B站服务器的反爬虫机制之一，**作为“门外汉”，还望各位大佬赐教**。

而在综合模式下——即先进行一次热度爬取、再进行一次时间爬取，理论上可以得到20000条评论左右，但因为B站神奇的热度排序机制，部分明明发表时间很新的评论在热度排序下也“名列前茅”，导致两次爬取的结果不可避免地存在大量重复，一般保底能获取15000条不重复评论，而能获取超过20000条不重复评论的情况并不多见；

### Q: 那有些视频（比如一些二次元游戏的重大版本PV），我预计其评论很可能超过2万条，能通过反复爬取评论来实现评论的完全爬取吗？

A: 是的！这就是迭代模式的原理——通过反复的爬取、多次的合并去重来突破单次爬取数量的上限！

那问题来了，我要在什么时候停止爬取呢？

一个简单的思路是限定迭代的时间，比如我预估这个新PV可能在24小时内突破3万评论的大关，而在24小时候其评论数量会大大放缓，那我就可以限定爬取时间为24小时，以这24小时的评论舆情来估计整体的评论舆情；

而另一个思路就是限定重复率——每次热度爬取或者时间爬取完成后，就和上一次爬取比较一下，看一下评论的重复率，如果重复率特别高，高到超过阈值（比如计算得到这一次和上一次爬取的重复率是100%，说明在爬取的这段时间内没有新评论出现），那就可以停止爬取；

以上两个方式简称“限定时间迭代模式”和“限定重复率迭代模式”，需要强调的是，**请提前给硬盘留好充足的空间，并且尽可能保证爬取时设备不会出现异常状况**；

### Q: 脚本是如何实现重复评论去重的？拖尾图又是如何生成的？

A: 在对响应数据进行分析时，笔者发现了每个评论都携带的两个特别的字符串：**rpid和parent**；

在测试后，笔者确信，**rpid是每个视频下评论的“唯一标识符”**，不随时间、点赞数或回复数的变化而变化，**rpid即为“Reply ID”**；而只要两条评论的rpid相同，就可以视为相同评论进行去重；

而parent就更加特别一点，以下面这张示例图为例：



- A是直接对视频进行评论，没有回复任何人，**A的评论的parent值为空**；

- B则是对A进行了回复，故**B的评论的parent值**和**A的评论的rpid值一致**；

- C则是对B进行了回复，故**C的评论的parent值**和**B的评论的rpid值一致**；同时，A不会收到C对自己评论进行回复的通知，因为parent值是唯一的；

通过这种parent-rpid配对方式，就可以厘清所有评论的回复关系——比如，这里B发表了多条评论，而C回复了其中一条评论；如果在网页端看的话，只能看到C在”@B“，无法看到C具体在回复B的哪一条评论，但在拖尾图中就相当直观了。

### 3、楼中楼拖尾文件功能

**功能说明**：

- 实际上，是现有这个独立脚本，再把该脚本的功能集成到主脚本中的……
- 换句话说，该脚本的一切功能，主脚本都可以实现
- 自动识别CSV数据中回复数超过5条的主楼评论
- 通过rpid和parent字段构建完整的评论层级关系树
- 生成可视化的评论树状结构图片和整合的Markdown文档
- 支持多层级嵌套显示，每个层级使用不同颜色区分

**实现逻辑**：
1. **数据解析**：从包含rpid和parent字段的CSV文件中提取评论关系数据
2. **树结构构建**：使用CommentTreeBuilder类根据parent-child关系构建评论树
3. **层级分析**：递归统计每个主评论的子回复数量，筛选出超过5条回复的评论
4. **可视化生成**：
   - 图片生成：使用PIL库创建带颜色层级的树状结构图
   - 文本生成：创建包含完整评论树的Markdown文档
5. **文件输出**：按照标准命名格式保存图片和文档文件

**功能效果**：
- **层级可视化**：清晰展示楼中楼回复的嵌套关系，每个层级使用不同颜色标识
- **内容完整性**：保留完整的评论内容，包括用户名、发布时间、点赞数等信息
- **结构清晰**：通过缩进和连接线展示父子评论关系
- **批量处理**：一次性处理所有符合条件的评论，生成统一的整合文档

**使用方法**：
楼中楼拖尾功能已集成到主程序中，无需单独调用：

1. **自动触发**：在主程序完成评论爬取并生成包含rpid和parent字段的CSV文件后自动执行
2. **生成条件**：仅对回复数超过5条的主楼评论生成拖尾文件
3. **文件位置**：生成的文件保存在主输出文件夹中

**输出文件**：
- `楼中楼拖尾图片_{视频标题}_{BV号}_{时间戳}_{评论ID}.png` - 单个评论的树状结构图
- `楼中楼拖尾整合_{视频标题}_{BV号}_{时间戳}.md` - 所有符合条件评论的整合文档

- **动态布局**：根据评论内容长度和层级深度自动调整图片尺寸
- **颜色编码**：不同层级使用不同颜色，提高可读性
