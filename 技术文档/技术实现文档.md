[bootstrap.check_and_install_dependencies()](../FuckBilibiliComments/bootstrap.py)

## 3. Cookie 与 API 签名

### 3.1 Cookie 输入与存储

- 用户用浏览器 Cookie-Editor 扩展导出整段 `key=value;key=value`
- [cookie.parse_cookie_editor_export](../FuckBilibiliComments/cookie.py) 解析为 dict
- [cookie.build_cookie_string](../FuckBilibiliComments/cookie.py) 按 `REQUIRED_COOKIE_KEYS` 白名单
  过滤，只保留：
  - `SESSDATA` / `bili_jct` / `DedeUserID` / `DedeUserID__ckMd5` — 登录态与 CSRF
  - `buvid3` / `buvid4` / `buvid_fp` / `buvid_fp_plain` / `_uuid` / `fingerprint` — 设备指纹
  - `b_nut` / `sid` / `b_lsid` — 会话标识
  - `bili_ticket` / `bili_ticket_expires` — 新版鉴权
  - `rpdid` — 跟踪 ID
- 重组后写入 [config.json](../config.json) 的 `accounts[i].cookie` 字段

### 3.2 User-Agent

- 提供 Firefox（默认）/ Chrome / 自定义 三选一
- 常量 `UA_FIREFOX` / `UA_CHROME` 定义在 [cookie.py](../FuckBilibiliComments/cookie.py)

### 3.3 w_rid 签名

[api.generate_w_rid(params)](../FuckBilibiliComments/api.py)：
1. 固定 salt `a = "ea1db124af3c7062474693fa704f4ff8"`
2. 按 `[oid, type, sort, ps, (pn|seek_rpid), plat, web_location, wts]` 顺序拼接 `k=v&...`
3. MD5(拼接串 + salt) 取 32 位 hex

### 3.4 封禁处理

收到 HTTP 412 等异常 → 抛 [CookieBannedException](../FuckBilibiliComments/errors.py) →
[cookie.try_switch_to_next_cookie](../FuckBilibiliComments/cookie.py) 按 `selected_account_index`
轮转到下一个 → 重启爬取。所有 cookie 都被封禁则终止。

## 4. 三种爬取模式

| 模式 | 适用场景 | 上限 | 策略 |
|------|---------|------|------|
| **测试** | 调试、抽样 | 用户指定页数（1–50） | 单次按热度或时间爬指定页数即停 |
| **综合** | 一般爬取 | ~15000 条 | 先按热度爬完，再按时间爬补充，最后基于 `rpid` 合并去重 |
| **迭代** | 新发布的高热度视频（>2 万评论） | 受限于时间或重复率 | 反复交替热度/时间爬取，多轮合并去重，直到时长用尽或重复率超阈值 |

### 4.1 综合模式流程

1. `crawl_all_comments_with_reason(mode=1)` 热度爬取，直到 API 返回无更多或结束原因触发
2. 若未爬完，`crawl_all_comments_with_reason(mode=0)` 时间爬取补充
3. [processing.merge_and_deduplicate_comments](../FuckBilibiliComments/processing.py) 基于 `rpid` 合并，
   重复评论保留爬取时间更新的版本
4. `process_and_organize_data` 双重整理：按热度排序输出主结果 CSV + 按时间统计

### 4.2 迭代模式流程

- **限定时间**：用户指定总爬取小时数 → 进入主循环，每轮内执行热度→时间各一次，
  把单轮结果写入 `原始数据/{热度,时间}爬取原始数据/迭代N_*.csv`，到点退出
- **限定重复率**：每轮结束计算与上一轮的 `rpid` 重复率，若超热度阈值或时间阈值则停
- 退出后 [processing.perform_iteration_deduplication](../FuckBilibiliComments/processing.py)
  对所有迭代轮次的数据做迭代去重，同 `rpid` 保留时间最新版本
- 额外生成 `迭代统计报告.txt`，分析迭代效率（重复率/新增评论数趋势）

### 4.3 重复评论识别

- 主键：`rpid`（B 站给每条评论分配的唯一标识）
- 同 `rpid` 出现多次时 → 保留 `爬取时间` 最大的版本
- 拒绝改的字段：发布时间、用户名、评论内容
- 动态字段：点赞数、回复数、`爬取时间`（每轮都更新）

## 5. 评论数据处理

### 5.1 回复关系树（楼中楼）

每条评论携带 `rpid` 与 `parent`：
- `parent == 0` 或缺失 → 主楼评论
- `parent == X.rpid` → 该评论是 X 的直接回复

[tree.CommentTreeBuilder](../FuckBilibiliComments/tree.py) 实现：

```
A (主楼)
├── B (B.parent == A.rpid)
│   ├── F (F.parent == B.rpid)
│   └── G
│       └── I (I.parent == G.rpid)
├── C
├── D
│   └── H
└── E
```

可视化输出：
- 节点显示为 `{用户名}：{评论内容}`
- 每层颜色不同（基于深度的 HSV→RGB 映射）
- **仅对回复数 > 5 的主楼**生成单独的拖尾图 PNG + 整合 md
- PIL 绘制；图片宽度根据节点字符数动态计算

### 5.2 按时间统计（粒度自适应）

[stats.generate_restructured_time_statistics](../FuckBilibiliComments/stats.py)：

1. 取所有评论中最大时间戳，计算与视频发布时间的差 Δt 和年/月/日跨度
2. 根据跨度决定输出粒度组合：

| 跨度 | 输出粒度 |
|------|---------|
| 跨年 | 年 + 月 + 日 |
| 同年跨月 | 月 + 日 |
| 同年同月跨日 且 Δt > 604800s（一周） | 日 |
| 同年同月跨日 且 Δt ≤ 604800s | 日 + 小时 |
| 同日 | 小时 + 分钟 |

3. **分钟/小时统计**：以"视频发布时间"为零点划区间（**不是整小时整分钟**）
   - 视频发于 22:27 → 第一段 22:27–23:27，**不是** 22:00–23:00
   - 区间宽度：分钟 60s，小时 3600s
4. **天/月/年统计**：直接按"年/月/日"键分组计数（避免误匹配评论文本里的"2025/8"等字符）
5. 每个粒度输出：
   - `txt` 统计报告（首尾时间、累计段数、总评论数、均值/峰值/谷值、逐段明细）
   - `png` 趋势折线图，标题：`{视频标题} - {BV号} - 评论数量变化趋势\n视频发布时间：{...} - 最新评论时间：{...} - 按{粒度}统计`

### 5.3 时间精细统计工具（独立脚本）

[tools/评论时间精细统计工具.py](../tools/评论时间精细统计工具.py)：在已有 CSV 上让用户**手动选定时间段**做更精细统计。

交互逻辑：
1. 输入 CSV 路径，校验"评论发布时间"列存在
2. 若最新评论与视频发布在同一日 → 提示"已达最大精度"，终止
3. 询问起止时间（年→月→日），自动跳过同年/同月的冗余输入
4. 校验：起 ≥ 视频发布日，止 ≤ 最新评论日；日期合法（含闰年校验）
5. 询问"起始时间是否与视频发布一致"：是 → 输出按"视频发出后 N 分钟"描述；否 → 按"起始时间后 N 分钟"描述
6. 以分钟、小时粒度统计 + 折线图

## 6. 输出文件夹结构

### 6.1 综合模式

```
评论爬取_{视频标题}_{BV号}_{HHMMSS}_{YYYYMMDD}/
├── logs/
│   ├── {timestamp}_bilibili_crawler_{bv}_main.log
│   └── {timestamp}_page_{N}_{bv}.log         （每页一份）
├── 原始数据/
│   ├── 评论爬取原始数据_热度爬取_*.csv
│   ├── 评论爬取原始数据_时间爬取_*.csv
│   ├── 评论爬取原始数据_合并去重_*.csv
│   └── 评论爬取原始数据_重复评论_*.csv
├── 按时间统计/
│   ├── 评论爬取统计结果_按小时统计_*.txt + .png
│   ├── 评论爬取统计结果_按日统计_*.txt + .png
│   ├── 评论爬取统计结果_按月统计_*.txt + .png
│   └── 评论爬取统计结果_按年统计_*.txt + .png
├── 评论爬取_按热度排序_*.csv                  （最终主结果）
├── 评论爬取_按热度排序且无楼中楼_*.csv         （仅主楼）
├── 评论爬取_统计数据_*.txt                    （爬取率、用户分布等）
├── 楼中楼拖尾图片/                            （回复数>5 才有）
│   └── 楼中楼拖尾图片_*.png
├── 楼中楼拖尾整合_*.md
└── 评论爬取_文件夹结构_*.md
```

### 6.2 迭代模式

与综合模式基本一致，差异：

```
└── 原始数据/
    ├── 热度爬取原始数据/
    │   └── 迭代1_*.csv, 迭代2_*.csv, ...      （每轮临时文件）
    ├── 时间爬取原始数据/
    │   └── 迭代1_*.csv, ...
    ├── 评论爬取原始数据_热度爬取（已迭代去重）_*.csv
    ├── 评论爬取原始数据_时间爬取（已迭代去重）_*.csv
    ├── 评论爬取原始数据_合并去重_*.csv
    ├── 评论爬取原始数据_重复评论_*.csv
    ├── 按热度迭代统计报告_*.txt
    ├── 按时间迭代统计报告_*.txt
    └── 迭代统计报告_*.txt                     （迭代效率分析）
```

### 6.3 文件命名规范

固定格式：`{文件前缀}_{视频标题}_{BV号}_{HHMMSS}_{YYYYMMDD}.{ext}`

- 视频标题先用 [io_utils.generate_safe_filename](../FuckBilibiliComments/io_utils.py)
  清洗（去除 `/\:*?"<>|` 等非法字符，过长截断）
- 时间戳保证同名文件不冲突
- BV 号便于后续脚本反解视频

## 7. 迭代模式后处理（设计中，未实现）

适用场景：迭代模式中途崩溃后只剩 `原始数据/{热度,时间}爬取原始数据/` 两个文件夹，
需要补完去重、统计、报告流程。

设计要点：
1. **BV 号从路径反推**：扫描传入路径，匹配 `BV` 后跟 10 位字母数字
2. 通过 B 站官方 API 用 BV 号补查视频元信息
3. 主输出目录命名为 `迭代模式后处理_{视频名称}_{BV号}`
4. 调用 `perform_iteration_deduplication` 完成去重
5. 输出与正常迭代模式一致的最终数据 + 时间统计 + 结构 md
6. 文件名加 `后处理_` 前缀以区分

## 9. 已知约束与坑

| 项 | 说明 |
|----|------|
| **单次爬取上限** | ~1 万条后 B 站会触发 `Connection aborted` 等连接级限制；综合模式靠两轮叠加突破到 ~1.5 万 |
| **412 封禁** | 爬取过频触发；脚本自动切下一个 cookie；不建议共享 cookie |
| **延时** | 经测试 3000ms 是稳定性与速度的最佳平衡；< 1000ms 容易触发封禁 |
| **B 站热度排序不稳定** | 同一视频两次按热度爬取，结果顺序会变；这就是为什么综合模式需要二次合并去重 |
| **动态/番剧不支持** | API 接口不同，未实现 |
| **手机短链不支持** | 短链需要重定向解析；脚本不识别 |
