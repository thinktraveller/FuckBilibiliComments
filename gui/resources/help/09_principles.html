工作原理

了解工具的底层运作机制，有助于理解为什么需要 Cookie、为什么存在爬取上限、以及输出文件里每一列数据的来源。

脚本在做什么？

本工具的本质是模拟已登录用户在浏览器里翻看评论的行为。

每次你在 B 站滚动评论区，浏览器就会向 B 站的评论 API 发送一次 HTTP GET 请求，B 站返回当页的评论数据（JSON 格式）。本工具通过 Python 的 requests 库不断重复这个过程，每页 20 条，直到所有评论页被爬取完毕。

这也是爬取单次存在上限的原因——B 站在连续请求约 500 页（10000 条）后会触发服务器限制，拒绝后续请求。

发给 B 站的请求

每次请求由两部分组成：

请求头（模拟浏览器身份）

字段内容作用
Cookie 你的 B 站登录凭证（含 SESSDATA 等字段） 告诉 B 站"我已登录，可以查看全部评论"。缺少此字段则只能看到少量评论
User-Agent 浏览器标识字符串 告诉 B 站"我使用的是某某浏览器"。与 Cookie 来源浏览器保持一致可降低被识别为异常请求的概率
Referer https://www.bilibili.com/ 告诉 B 站"这个请求来自 B 站页面内部"，模拟正常页面跳转行为

请求参数（告诉 B 站要哪个视频的第几页）

参数名示例值含义
oid 115066334743950 视频的 AV 号（av id），唯一标识视频。工具从 BV 号自动转换获取
type 1 资源类型，1 表示普通视频评论
sort 0 或 1 排序方式：0 = 热度排序（综合热度算法），1 = 时间排序（最新优先）
pn 1、2、3… 页码（Page Number），工具从第 1 页开始逐页请求
ps 20 每页评论数（Page Size），B 站固定每页返回 20 条评论

B 站返回的评论字段

B 站以 JSON 格式返回当页评论列表。每条评论包含以下关键字段，工具提取后保存到输出 CSV 文件：

JSON 字段输出 CSV 列名说明
rpid_str rpid 评论唯一 ID（Reply ID）。在同一视频下永久不变，不随点赞数、回复数变化而改变。去重的唯一依据
parent_str parent 被回复评论的 rpid。主楼评论此值为 0；楼中楼评论此值等于它所回复的那条评论的 rpid
content.message 评论内容 评论正文文本
member.uname 用户名称 评论者昵称
member.mid 用户 ID 评论者的 B 站 UID（数字）
member.level_info.current_level 用户等级 评论者当前 B 站等级（0–6）
member.sex 性别 评论者性别（B 站注册时填写的信息）
like 点赞数 该评论获得的点赞数量（爬取时刻的快照值）
rcount 回复数 该评论的楼中楼回复总数（爬取时刻的快照值）
ctime 时间戳 / 发布时间 评论发布时间的 UNIX 时间戳（秒）；工具同时保存原始时间戳和可读格式（如 2024-01-15 14:30:22）
reply_control.location IP 地区 评论者发布时的 IP 属地（如"广东"）；B 站于 2022 年起在评论中展示此信息

rpid 和 parent 字段的作用

rpid → 去重依据

只要两条评论的 rpid 相同，无论点赞数和回复数如何变化，都判定为同一条评论并去重。这是综合模式两次爬取合并去重、以及 CSV 去重工具的底层逻辑。

parent → 层级关系依据

通过逐层匹配"某条评论的 parent = 另一条评论的 rpid"，可以还原出完整的评论层级树。主楼评论的 parent 为 0，楼中楼评论的 parent 指向它直接回复的那条评论。这是生成楼中楼拖尾图的核心依据。

为什么最终 CSV 文件没有 rpid 列？

输出文件夹根目录下的"最终数据 CSV"是给人阅读用的，已将 rpid 和 parent 列移除以保持简洁。需要进行 CSV 去重合并时，请使用"原始数据"子目录下的"合并去重结果"文件，该文件保留了 rpid 列。
