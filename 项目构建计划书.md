# 项目构建计划书 · FuckBilibiliComments GUI 化

> 版本：v1.0
> 日期：2026-05-18
> 目标：在保留现有 CLI 入口的前提下，为 FuckBilibiliComments 增加一套 Windows 桌面 GUI，覆盖评论爬取、时间统计、CSV 去重、账号管理与历史记录等全部能力。

---

## 1. 项目概述

### 1.1 项目名
**FuckBilibiliComments**（开发代号；正式名称可在 M5 阶段再定）。

### 1.2 项目目标
将现有的命令行工具（CLI）和两个独立 Python 脚本（`tools/`）整合为一个面向 Windows 桌面用户的 GUI 程序，使非技术用户也能：

1. 通过界面输入 BV 号 / URL 即可启动评论爬取
2. 可视化管理多个 B 站账号 Cookie / UA
3. 实时观察爬取进度，可随时中止、可同时跑多个任务
4. 一键复用现有的时间统计、CSV 去重工具
5. 通过历史记录回溯过往任务，快速重爬或进入分析

### 1.3 用户画像

| 维度 | 描述 |
|------|------|
| 主用户 | 项目作者本人（非计算机专业，靠 AI 协助开发）|
| 次用户 | 不会用命令行的同好 / 普通 B 站重度用户 |
| 技术水平 | 能打开 .exe、能装浏览器扩展，但不熟悉 Python / 终端 |
| 使用场景 | Windows 11/10 桌面，单人本地使用 |

### 1.4 核心价值
- **零命令行门槛**：从 cmd → 双击 exe
- **零脚本切换成本**：原本散落在 `tools/` 的两个工具收纳进同一界面
- **可观察、可干预**：实时日志 + 中止 + 多任务并行
- **可回溯**：历史记录支持复盘、重爬、跳转分析

### 1.5 与现有 CLI 的关系定位
本期 **GUI 与 CLI 并行维护**：

- CLI 入口 `FuckBilibiliComments.py` 完整保留，行为不变
- GUI 与 CLI 共享同一套核心业务代码（`crawl/processing/stats/reports/tree` 等）
- 任意一方修复 bug、新增能力，另一方应自动受益
- 不存在"GUI 版"和"CLI 版"两份独立逻辑的代码分叉

## 2. 技术选型与理由

### 2.1 关键决策一览

| 项目 | 选型 | 备选 | 选择理由 |
|------|------|------|---------|
| GUI 框架 | **PySide6** | Tkinter / Flet / PyQt6 / Electron | 见 2.2 |
| 界面设计工具 | **Qt Designer**（.ui 文件） | 手写代码 | 拖拽式设计，非专业用户友好；与 PySide6 官方配套 |
| 异步方案 | **QThread + 信号槽** | asyncio / threading | Qt 原生方案，UI 线程零阻塞；信号槽天然线程安全 |
| 打包工具 | **PyInstaller** | Nuitka / cx_Freeze | 社区成熟、坑少、文档多、AI 资料丰富 |
| 打包模式 | **onedir**（发布） | onefile | 启动快、杀软误报率低、便于热更新单个文件 |
| 包管理 | 沿用 **bootstrap.py** | poetry / pdm | 维护一致性；新增 `requirements-lock.txt` 锁版本 |
| Python 版本 | **3.9+**（推荐 3.11） | 3.7 / 3.12 | PySide6 6.6+ 要求 3.9+；3.11 性能最佳 |
| 历史记录存储 | **JSON 文件** | SQLite | 数据量小（千级条目）、零依赖、人类可读、便于备份 |

### 2.2 为何选 PySide6 而非其他

| 候选 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| Tkinter | 标准库零依赖 | 控件丑陋、无 Tab 美化、表格能力弱、并发体验差 | ❌ 不适合本项目复杂度 |
| Flet | 现代化、Flutter 风格 | 仍在快速迭代、桌面打包链不稳、中文社区资料少 | ❌ 维护风险高 |
| PyQt6 | 功能等同 PySide6 | GPL 协议，商业分发需付费 | ❌ 协议风险 |
| Electron | 前端生态丰富 | Node + Python 双栈、打包体积 100MB+、内存占用大 | ❌ 维护门槛高 |
| **PySide6** | LGPL 协议、Qt 官方维护、Qt Designer 拖拽设计、控件齐全（QTreeView/QTableView/QProgressBar/QPlainTextEdit 全部开箱即用） | 安装包略大（~80MB） | ✅ **最佳选择** |

### 2.3 异步方案说明

**核心约束**：用户要求"爬取过程中允许操作其他面板"（不锁定界面 + 多任务并发）。

**方案**：
- 每个爬取任务 = 一个 `QThread` 子类，构造时注入 `task_id`、`params`
- 核心业务通过 `pyqtSignal(str)` / `pyqtSignal(int, int)` 把日志、进度推回主线程
- 主线程仅做信号路由（哪条日志推到哪个 Tab 的日志窗），不做任何阻塞计算
- 中止按钮通过 `QThread.requestInterruption()` + 业务层定期检查 `isInterruptionRequested()` 实现优雅退出

### 2.4 依赖清单（计划版本约束）

```text
# 业务相关（沿用现有）
requests>=2.31
pandas>=2.0
matplotlib>=3.7

# GUI 新增
PySide6>=6.6,<6.8

# 打包（仅开发依赖）

```

### 3.2 关键架构约束

| 约束 | 描述 |
|------|------|
| **核心业务 UI 无关** | `crawl.py` 等核心模块禁止 `import PySide6`，禁止 `print` / `input`，全部通过回调暴露 |
| **回调三件套** | 统一接口：`log_callback(level, msg)` / `progress_callback(current, total)` / `interrupt_check() -> bool` |
| **同接口双实现** | CLI 注入"打印到 stdout"的回调；GUI 注入"emit 信号"的回调 |
| **配置单一来源** | `config.json` 是账号配置唯一来源；GUI 改完即写盘，CLI 重启即生效 |
| **服务层无状态** | services/ 内函数尽量纯粹，状态归 task_manager / history 集中管理 |

| 维度 | 设计 |
|------|------|
| 并发数 | 无硬上限，建议同 BV 不重复启动；不同任务可任意并行 |
| 线程模型 | 每任务一 QThread，互不影响 |
| 日志隔离 | 各 Tab 维护自己的 QPlainTextEdit；同 Tab 内多任务通过"任务下拉切换"显示 |

| 状态汇总 | 底部任务状态栏从 task_manager 拉数据，每 500ms 刷新 |

## 5. 核心模块改造方案

### 5.1 回调接口抽象（新增 `callbacks.py`）

```python
# FuckBilibiliComments/callbacks.py 设计草案
from dataclasses import dataclass
from typing import Callable

@dataclass
class TaskCallbacks:
    log:        Callable[[str, str], None]       # (level, message)
    progress:   Callable[[int, int], None]       # (current, total)
    prompt:     Callable[[str], str]             # 询问用户（CLI: input, GUI: QInputDialog）
    is_aborted: Callable[[], bool]               # 检查中止信号

# CLI 默认实现
def make_cli_callbacks() -> TaskCallbacks: ...

# GUI 实现由 worker 注入（emit 信号）
```

### 5.2 各模块改造点

| 模块 | 改造点 | 是否破坏 CLI |
|------|--------|--------------|
| `cli.py` | 把"流程编排"挪到 `services/crawl_service.py`，本文件仅做参数解析 + 调用服务 | 不破坏（功能等价） |
| `crawl.py` | 移除所有 `print`，改为 `cb.log(...)`；循环内每页结束检查 `cb.is_aborted()` | 不破坏 |
| `processing.py` | 同上 | 不破坏 |
| `stats.py` | 把 `tools/评论时间精细统计工具.py` 的小时/分钟级算法合并进来作为函数 | tools 脚本改为薄壳 |
| `reports.py` / `tree.py` | 把生成路径作为返回值返回（而非仅打印），方便 GUI 收集 | 不破坏 |
| `main.py` | 保留为 CLI 主流程封装 | 不破坏 |
| `tools/*.py` | 改为 10 行内的薄壳，仅做 `argparse + services 调用` | 行为不变 |

### 5.3 改造顺序保护 CLI 不破

1. 先添加 `callbacks.py` + CLI 默认实现
2. 改造一个模块（如 `crawl.py`），用 CLI 跑通完整流程作为回归测试
3. 通过后再改下一个；每次提交都保证 `python FuckBilibiliComments.py` 可正常运行
4. 完成所有改造后再开始写 GUI

## 7. 任务管理设计

### 7.1 task_manager 职责

- 注册任务：`register(worker) -> task_id`
- 查询任务：`list_running()` / `get(task_id)`
- 中止任务：`abort(task_id)`
- 状态变更通知：发出全局信号 `task_state_changed(task_id, new_state)`
- 与 history_service 联动：任务结束自动写历史记录

### 7.2 QThread Worker 基类

```python
# gui/workers/base_worker.py 设计草案
class BaseWorker(QThread):
    log_signal      = Signal(str, str)
    progress_signal = Signal(int, int)
    finished_signal = Signal(str, bool, str)   # task_id, ok, msg

    def __init__(self, task_id, params): ...
    def request_stop(self): self._stop_flag = True
    def is_aborted(self) -> bool: return self._stop_flag
    def run(self):  # 子类实现，构造 TaskCallbacks 并调用 services
        ...
```

### 7.3 并发安全要点

- `history.json` 写入使用 `QMutex` 保护
- `config.json` 读改写采用"读 → 内存改 → 写"，并提供 GUI 主线程独占编辑（账号管理 Tab 打开时其他任务读取最新副本即可）
- B 站接口请求方面，建议同账号同视频不并发（在 task_manager 启动前做去重检查并提示）
